# ai-steward

Telegram-бот с доступом к AI CLI. Версия 2.0.0.

## Стек

- **Python 3.11+**, пакетный менеджер **uv**
- **aiogram 3.15** — Telegram Bot API (asyncio)
- **Claude Code CLI** — вызов через subprocess (не API)
- **faster-whisper** — STT (речь → текст)
- **edge-tts** — TTS (текст → речь)
- **pytesseract** — OCR (фото → текст)
- **pydantic-settings** — конфигурация через `.env`

## Структура

```
src/ai_steward/
├── __main__.py          # Точка входа
├── bot.py               # Фабрики create_bot, create_dispatcher
├── config.py            # Settings (pydantic-settings)
├── state.py             # AppState (in-memory)
├── keyboards.py         # Inline-клавиатуры с пагинацией
├── handlers/
│   ├── __init__.py      # Общие хелперы (send_long, send_files)
│   ├── commands.py      # /help, /status, /new, /model, /voice, /cancel
│   ├── text.py          # Текстовые сообщения
│   ├── voice.py         # Голосовые сообщения
│   ├── photo.py         # Фотографии
│   └── document.py      # Документы
├── services/
│   ├── claude.py        # run_claude() → subprocess CLI
│   ├── storage.py       # SessionStorage (JSON persistence)
│   ├── speech.py        # STT + TTS
│   ├── ocr.py           # OCR через tesseract
│   └── format_telegram.py  # Markdown → Telegram HTML
└── middlewares/
    └── auth.py          # Авторизация, роли, лимиты
```

## Соглашения

- Отступы: **4 пробела**
- Комментарии: на **русском**
- Типизация: **type hints** везде
- Логирование: `logging.getLogger(__name__)`
- Роутеры: `text.router` регистрируется **последним** (catch-all)
- Зависимости пробрасываются через `AuthMiddleware` → `data["settings"]`, `data["state"]`, `data["storage"]`

## Запуск

```bash
make install   # uv sync
make run       # uv run ai-steward
```

## Документация — правила обновления

### CHANGELOG.md

Формат: [Keep a Changelog](https://keepachangelog.com/ru/1.1.0/).

#### Триггеры обновления

| Событие | Секция | Пример |
|---------|--------|--------|
| Новая функция | `Added` | Добавлена команда /export |
| Изменение поведения | `Changed` | Формат ответа изменён на HTML |
| Исправление бага | `Fixed` | Исправлен сброс счётчика лимитов |
| Удаление функции | `Removed` | Удалена команда /old |
| Устаревание | `Deprecated` | Команда /legacy помечена как устаревшая |
| Безопасность | `Security` | Исправлена валидация user_id |

#### Правила

1. Записывать в секцию `[Unreleased]`
2. Описывать **для пользователя** (что изменилось), а не для разработчика (какой код)
3. Одна строка = одно изменение
4. При релизе — перенести из `[Unreleased]` в новую версию с датой

#### Формат записи

```markdown
### Added

- Команда `/export` для экспорта истории сессий в JSON
```

### ADR (Architecture Decision Records)

Директория: `docs/adr/`. Индекс: `docs/adr/README.md`.

#### Когда создавать

1. **Новая зависимость** — добавление библиотеки, фреймворка, внешнего сервиса
2. **Смена архитектуры** — изменение структуры модулей, хранилища, потока данных
3. **Замена технологии** — переход с одного инструмента на другой (статус старого ADR → `Заменено`)

#### Когда НЕ создавать

1. Мелкие рефакторинги без смены архитектуры
2. Баг-фиксы
3. Обновление версии зависимости (без смены самой зависимости)

#### Шаблон

```markdown
# ADR-NNN: Название

**Статус:** Принято
**Дата:** YYYY-MM-DD

## Контекст
Проблема и условия (3-5 предложений).

## Решение
Что выбрали и ключевые аргументы (3-5 пунктов).

## Альтернативы
- **Вариант A** — причина отказа.
- **Вариант B** — причина отказа.
```

#### После создания ADR

Обновить таблицу в `docs/adr/README.md` — добавить строку с номером, названием, статусом и датой.

## Ограничения

- **Не читать `.env`** — содержит секреты. Использовать `.env.example` как справку.
- **Не пушить** — только локальные коммиты.
- **Только `$PWD`** — изменять файлы только в текущей директории проекта.
