# REQ-002: Требования для TASK-002

## Task
TASK-002: Система планирования и отслеживания дня

## Статус
Approved

## Архитектурное решение
Вариант A — единая модель. Уведомления и задачи объединены в одну сущность `PlanItem`.
Файл `notify.json` переименовывается в `planner.json`. Миграция автоматическая при загрузке.

## Функциональные требования (FR)

| ID    | Требование | Приоритет |
|-------|-----------|-----------|
| FR-1  | **Единая модель PlanItem** — заменяет текущий `Notification`. Поля: id, title, description, category (str: medication/event/task/block/reminder/...), priority (high/medium/low/none), date (date привязки к дню), time_start (HH:MM, опц.), time_end (HH:MM, опц.), deadline (datetime, опц.), remind_before (list[int], минуты), repeat (RepeatRule, опц.), recipients (list[int]), status (pending/in_progress/done/skipped/cancelled), created_at, created_by, sent_reminders | Must |
| FR-2  | **Файл planner.json** — заменяет `notify.json`. Корневая структура `PlannerFile` с полем `items: list[PlanItem]`. Хранится в директории проекта (как текущий notify.json) | Must |
| FR-3  | **Автоматическая миграция** — при загрузке: если существует `notify.json`, конвертировать записи `Notification` → `PlanItem`, сохранить как `planner.json`, удалить (или переименовать) старый файл | Must |
| FR-4  | **Сервис planner_service** — заменяет `notification_service`. CRUD: load, save, add, remove, update, get_by_date(date), get_active, is_due, mark_sent. Функция `find_conflicts(items, new_item) → list[PlanItem]` — проверка пересечений по time_start/time_end. Та же логика проверки времени и отправки напоминаний | Must |
| FR-5  | **Команда /plan** — показать план на сегодня. Группировка: сначала элементы с time_start (хронологически), затем без времени (по приоритету). Inline-кнопки навигации под сообщением: `◀ Вчера`, `Завтра ▶`, `Послезавтра ▶▶`, `📅 Неделя`. Кнопки переключают отображаемую дату (edit_text). Эмодзи статусов: ⬜ pending, 🔄 in_progress, ✅ done, ⏭ skipped, ❌ cancelled | Must |
| FR-6  | **Отображение прошедших элементов** — для элементов с датой/временем в прошлом: показывать финальный статус-пометку (✅ сделано / 📦 перенесено на DD.MM / ❌ отменено / ⬜ не выполнено) | Must |
| FR-7  | **Отображение будущих элементов** — для элементов с датой/временем в будущем: показывать время начала и длительность (если задан time_end, вычислить разницу). Формат: `10:00 (1ч 30мин)` | Must |
| FR-8  | **Создание через Claude с обнаружением конфликтов** — пользователь описывает план в чате, Claude формирует PlanItem, вызывает add(). Сервис записывает элемент и возвращает список конфликтов (find_conflicts). Если конфликты есть — Claude показывает предупреждение с вариантами решения (сдвиг, разделение, оставить как есть). Пользователь отвечает цифрой (1/2/3) или Да/Нет. Claude применяет выбранный вариант | Must |
| FR-9  | **Обновление статуса** — функции: mark_done(id), mark_skipped(id), set_status(id, status). Claude вызывает по запросу пользователя | Must |
| FR-10 | **Каждый PlanItem = уведомление** — каждый элемент плана автоматически работает как уведомление: отправка за remind_before минут ДО начала + отправка в момент time_start (remind_before=[0] всегда включён неявно). Используется существующая логика is_due/mark_sent | Must |
| FR-11 | **Утренний дайджест** — автоотправка обзора дня в конфигурируемое время. Содержит: задачи на сегодня, предстоящие события, перенесённые из вчера. Отправляется всем пользователям с ролью admin | Must |
| FR-12 | **Вечерний итог** — автоотправка итогов дня. Содержит: done/skipped/pending, completion rate (%). Конфигурируемое время | Must |
| FR-13 | **Перенос незавершённых задач** — при формировании утреннего дайджеста: pending-задачи вчерашнего дня копируются на сегодня с пометкой «перенесено» (поле carried_over: bool, carried_from: date) | Must |
| FR-14 | **Конфигурация** — новые поля Settings: `plan_morning_time` (HH:MM, default "08:00", "" = выкл), `plan_evening_time` (HH:MM, default "22:00", "" = выкл) | Must |
| FR-15 | **Переименование NotificationManager → PlannerManager** — обрабатывает и напоминания (remind_before), и дайджесты (утро/вечер) в едином цикле | Must |
| FR-16 | **/notify → алиас /plan** — обратная совместимость команды | Must |
| FR-17 | **Фильтрация /plan** — `/plan tasks`, `/plan events`, `/plan blocks` — фильтр по category | Must |

## Нефункциональные требования (NFR)

| ID     | Требование | Категория | Приоритет |
|--------|-----------|-----------|-----------|
| NFR-1  | **Миграция без потерь** — все существующие notify.json конвертируются в planner.json с сохранением всех данных. Поле date вычисляется из datetime | Reliability | Must |
| NFR-2  | **Обратная совместимость загрузки** — если planner.json не существует, но есть notify.json — мигрировать автоматически. Если ни того ни другого — пустой PlannerFile | Reliability | Must |
| NFR-3  | **Единый цикл сканирования** — PlannerManager обрабатывает remind_before и дайджесты без дублирования кода и без увеличения частоты циклов | Maintainability | Must |
| NFR-4  | **Время цикла сканирования** — не более 2x от текущего | Performance | Must |
