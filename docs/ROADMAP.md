# План развития проекта

## Этап 1: Модуль уведомлений

**Описание:** Модуль, который сканирует все проекты раз в 5 минут и отправляет уведомления пользователю в Telegram.

**Задачи:**
- Реализовать периодическое сканирование проектов (интервал 5 минут)
- Определить критерии для генерации уведомлений
- Отправка уведомлений через Telegram-бот
- Настройка включения/отключения уведомлений per-user

## Этап 2: Модуль приёма пересылок и произвольного контента

**Описание:** Модуль, который принимает пересылки из других чатов, файлы, сообщения, фото и прочий контент, не связанный с текущим проектом. Позволяет собирать и обрабатывать разнородную информацию из разных источников.

**Задачи:**
- Обработка пересланных сообщений (forwarded messages) из любых чатов
- Приём файлов, фото, документов, голосовых и прочих типов контента
- Хранение принятого контента независимо от активной сессии/проекта
- Классификация и организация входящего контента

## Этап 3: Резервное копирование и disaster recovery

**Описание:** Автоматическое резервное копирование всех пользовательских данных для быстрого восстановления бота на резервном сервере с минимальным простоем.

**Проблема:** При уничтожении текущего сервера теряются все данные пользователей — сессии, проекты, аналитика. Нужна возможность поднять бот на новом сервере за минимальное время.

**Данные для бэкапа:**
- `data/sessions.json` — метаданные сессий и проектов всех пользователей
- `data/analytics.db` — SQLite база аналитики
- Per-user `projects_dir` — рабочие директории проектов (основной объём данных)
- `.env` — конфигурация (бэкапится отдельно, вручную)

**Задачи:**
- Выбрать стратегию бэкапа (rsync на удалённый сервер / S3-совместимое хранилище / оба варианта)
- Реализовать инкрементальный бэкап projects_dir (полный бэкап каждый раз — дорого по трафику)
- Бэкап sessions.json и analytics.db по расписанию (cron или встроенный scheduler)
- Скрипт восстановления: разворачивает бот из бэкапа на чистом сервере (install deps + restore data + start)
- Мониторинг: уведомление в Telegram если бэкап не прошёл
- Документация: runbook по восстановлению (шаг за шагом)
- Тестирование: периодическая проверка что бэкап рабочий (restore dry-run)

**Целевые метрики:**
- RPO (допустимая потеря данных): ≤ 1 час
- RTO (время восстановления): ≤ 30 минут

## Этап 4: Персонализация общения под пользователя

**Описание:** Адаптация стиля общения Claude под каждого пользователя — возраст, пол, уровень технической грамотности, предпочтения в общении. Бот должен разговаривать с бабушкой иначе, чем с программистом.

**Задачи:**
- Расширить конфиг пользователя в `USERS` — добавить поля профиля (имя, возраст/поколение, пол, уровень, стиль общения)
- Формировать system prompt для Claude на основе профиля: обращение (ты/вы), сложность лексики, объём пояснений, примеры
- Команда `/profile` — пользователь может сам посмотреть и изменить свой профиль через inline-меню
- Предусмотреть пресеты: «молодой разработчик», «опытный специалист», «нетехнический пользователь», «пожилой человек» — с возможностью тонкой настройки
- Передавать профиль в system prompt при каждом вызове `run_claude()`

**Примеры адаптации:**
- Молодой разработчик: краткие ответы, сленг допустим, код без лишних пояснений
- Пожилой нетехнический пользователь: обращение на «вы», простой язык, пошаговые инструкции, терпеливые объяснения
- Опытный специалист: минимум воды, технические детали, без упрощений

## Этап 5: Просмотр Pull Requests из GitHub

**Описание:** Возможность просматривать PR из GitHub-репозиториев прямо в Telegram — список открытых PR, детали, диффы, комментарии, статусы проверок.

**Задачи:**
- Настройка GitHub-токена в конфиге пользователя (per-user, для доступа к приватным репозиториям)
- Команда `/pr <owner/repo>` — список открытых PR репозитория
- Просмотр деталей PR: описание, автор, ветка, статус CI, reviewers
- Просмотр диффа PR (с форматированием для Telegram)
- Просмотр комментариев и review-комментариев
- Inline-меню навигации: список PR → детали → дифф → комментарии
- Привязка репозитория к проекту — чтобы не указывать `owner/repo` каждый раз
- Уведомления о новых PR / изменениях статуса (интеграция с Этапом 1)

## Этап 6: Перенос claude-bot в рабочую директорию проектов

**Описание:** Исследовать возможность переноса кода бота из `/opt/claude_bot/` в `~/Henry_Bud_GitHub/claude-bot/` — туда, где находятся рабочие проекты. Это позволит редактировать код бота через Telegram (через самого бота), но создаёт риски.

**Вопросы для исследования:**
- Pros и cons переноса в рабочую директорию
- Риск случайного редактирования критичных файлов бота (bot.py, config.py, handlers/) через Telegram
- Риск остановки бота в процессе редактирования (синтаксическая ошибка → краш)
- Как перезапустить бота через Telegram, если он упал? (catch-22: бот не работает → не можешь отправить команду)
- Нужен ли watchdog/supervisor (systemd, supervisord) для автоперезапуска после краша?
- Нужен ли механизм отката (git reset) если редактирование сломало бота?
- Можно ли сделать hot-reload без полного перезапуска?
- Разделение прав: какие файлы можно редактировать через Telegram, какие — нет (whitelist/blacklist)?
