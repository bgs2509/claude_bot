# ADR-011: Единая модель PlanItem для задач и уведомлений

**Статус:** Принято
**Дата:** 2026-03-17
**Task:** TASK-002

## Контекст

При разработке системы планирования дня возник вопрос: хранить задачи и уведомления как отдельные сущности (два файла, два сервиса) или объединить их в одну модель. Существующая модель Notification покрывает только отправку напоминаний по расписанию и не имеет полей для статуса выполнения, приоритета и переноса задач. Нужно решить, как расширить хранилище данных, чтобы поддержать управление задачами дня без дублирования логики расписания.

## Решение

Выбран Вариант A — единая модель PlanItem, где каждая задача одновременно является потенциальным уведомлением.

Ключевые аргументы:
- Каждая задача дня может иметь напоминание — дублирование отдельной сущности Notification избыточно.
- Единый цикл сканирования (PlannerManager) обрабатывает и напоминания, и дайджесты, без координации между двумя менеджерами.
- Файл planner.json заменяет notify.json с автоматической бесшовной миграцией; данные не теряются.
- Поля `remind_before` и `sent_reminders` в PlanItem полностью покрывают функциональность старого Notification.
- Упрощается модель доступа: один сервис (PlannerService) вместо двух (NotificationService + отдельный TaskService).

## Альтернативы

- **Вариант B: Две модели, общий файл** — Task и Notification хранятся в одном JSON с разными типами. Отклонён: усложняет схему файла, требует union-типов при десериализации, не даёт реального выигрыша перед единой моделью.
- **Вариант C: Два файла, два сервиса** — tasks.json и notify.json остаются раздельными. Отклонён: дублирование логики расписания в двух менеджерах, сложная координация при переносе задач (перенесённая задача должна тянуть за собой уведомление).
- **Вариант D: Расширить Notification** — добавить в существующую модель поля статуса и приоритета. Отклонён: название Notification противоречит семантике «задача дня»; модель стала бы перегруженной без концептуальной чистоты.
