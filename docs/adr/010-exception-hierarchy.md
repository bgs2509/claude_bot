# ADR-010: Иерархия исключений с каталогом user-friendly сообщений

**Статус:** Принято
**Дата:** 2026-03-15

## Контекст

До рефакторинга ошибки обрабатывались разрозненно: каждый хендлер содержал собственные try/except с захардкоженными текстами ответов пользователю. `ErrorMiddleware` логировал все исключения одинаково — на уровне error со stack trace, не различая ожидаемые бизнес-ошибки и сбои инфраструктуры. User-facing тексты дублировались в нескольких местах.

## Решение

- Трёхуровневая иерархия: `AppError` (базовый) → `DomainError` (бизнес-логика, ожидаемые) и `InfrastructureError` (внешние сервисы, ФС, CLI).
- Каждое исключение несёт `user_message_key` — ключ в каталоге `USER_MESSAGES` (`errors.py`).
- `USER_MESSAGES` — единый словарь user-friendly сообщений (SSOT), поддерживает шаблоны с параметрами.
- `get_user_message(key, **kwargs)` — безопасное получение сообщения с fallback на `unexpected_error`.
- `ErrorMiddleware` различает типы: `DomainError` → warning без stack trace, остальные `AppError` → error со stack trace.

## Альтернативы

- **HTTP-коды (4xx/5xx)** — избыточно для Telegram-бота, не несёт ценности для пользователя.
- **Один класс `BotError`** — не даёт дифференциации при логировании, все ошибки трактуются одинаково.
- **gettext / i18n** — избыточно для бота с одной локалью (русский), усложняет поддержку без реальной выгоды.
